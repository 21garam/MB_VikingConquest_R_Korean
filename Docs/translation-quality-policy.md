# Viking Conquest 한글화 — 번역 퀄리티 정책

## 1. 목적
- 번역 일관성과 품질 유지
- 오역·누락·표기 통일을 위한 기준 제시
- 퀄리티 향상 작업 시 판단 기준 제공

## 2. 품질 기준

### 2.1 의미 정확성
- 원문의 의미를 정확히 전달할 것. **원문의 범위는 References/en 뿐만 아니라 References 내 다른 언어(cns, cnt, de, es, tr 등)를 모두 참고**하여, 여러 참조를 종합해 의미를 파악하고 번역한다.
- 게임/역사 맥락에 맞는 용어 사용 (예: Charge → 돌격, Banner/coat of arms → 문장·깃발)
- 플레이스홀더({s1}, {reg1} 등) 및 서식(%d, ^^ 등) 변경·삭제 금지

### 2.2 일관성
- 동일 개념은 동일 용어로 통일 (예: pening → 페닝, jarl → 야를)
- 인물별 말투·어투는 캐릭터별로 일관 유지 (README 반영)
- 고유명사(지명, 인명, 아이템명)는 프로젝트 내 기존 표기 참조

### 2.3 문체·가독성
- UI/메뉴: 짧고 명확한 문장
- 대화문: 시대감을 위한 어미·존칭 허용, 과도한 고어는 자제
- 띄어쓰기·붙여쓰기: 국립국어원 맞춤법 기준 (의존명사 띄어쓰기 등)

### 2.4 금지 사항
- 원문에 없는 내용 임의 추가
- 키(key) 변경 또는 삭제
- CSV 구분자(파이프 |) 및 이스케이프 규칙 위반

## 3. 점검 항목 (퀄리티 체크리스트)
- [ ] UI/전투 용어: Charge, Flip Horizontal/Vertical 등 게임 용어 정확성
- [ ] 숫자·날짜 서식: {reg1}, {reg2} 등 순서가 한국어 표현과 맞는지
- [ ] 대화문 말투: 인물별 일관성
- [ ] 미번역/빈 문자열 여부
- [ ] 오타·띄어쓰기 샘플 검수

## 4. 수정 승인 원칙
- **모든 번역 수정·추가는 담당자(또는 지정된 리뷰어) 승인 후 반영**
- 분석·제안 단계와 실제 파일 수정 단계를 구분할 것

## 5. 대량 수정 시 속도·효율
여러 줄을 한꺼번에 고칠 때는 **한 군데씩 열고 닫는 방식**보다 아래를 권장한다.

### 5.1 한 번 읽고, 한 번 쓰기
- 파일을 **한 번만** 읽고, 메모리(문자열)에서 **모든 수정**을 적용한 뒤, **한 번만** 저장한다.
- 수정 대상이 많을수록, “한 줄 수정 → 저장 → 다시 열기” 반복보다 훨씬 빠르다.

### 5.2 스크립트로 일괄 치환
- 수십~수백 군데를 고칠 때는, **한 번만** 파일을 열어서 스크립트가 전부 치환한 뒤 **한 번만** 저장하게 하는 것이 가장 빠르다.
- 예: Python 등으로 `파일 읽기 → replace(패턴1, 수정1); replace(패턴2, 수정2); … → 파일 쓰기`.
- 주의: “이스트 앵글리아”처럼 **건드리지 말아야 할 문자열**이 있으면, 더 긴/구체적인 문맥으로 치환하거나 예외 처리를 넣는다.

### 5.3 replace_all 활용
- **같은 문구**가 여러 줄에 똑똑 같을 때는, 치환 한 번에 `replace_all` 옵션으로 전부 바꾸면 호출 횟수를 줄일 수 있다.
- 문맥이 다른 경우(예: “앵글리아”만 바꿀 곳과 “이스트 앵글리아”는 유지할 곳)에는 패턴을 구체적으로 잡거나, 5.1·5.2 방식을 쓴다.
